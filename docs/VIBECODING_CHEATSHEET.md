# Vibecoding 规范速查手册

> 从 how-to-vibecoding 仓库提炼,供开发时随时参阅。
> 原仓库:~/how-to-vibecoding/

## 一、协同规矩(红线)

1. **动手前先给 3-7 步计划**,标注每一步怎么验证
2. **改完必须贴 diff**,不要只给总结
3. **收工前自觉跑 lint、test、build**;跑不了的说明原因
4. **任何情况都不许泄露 secrets**(.env、key、token、credentials)
5. **最小化改动**,不要为了美观重构大片代码
6. **修改方案做到幂等**:跑几次不产生重复追加

## 二、停手条件(Stop conditions)

遇到以下情况立刻停手汇报:
- 要引入新依赖或做大版本升级
- 改动超过 5 个文件
- 连续 2 次排错失败找不到原因
- 涉及数据库 schema 变更
- 涉及 API Key / 认证逻辑修改

## 三、需求描述模板

```text
## Context(背景)
(仓库定位 / 相关模块 / 限制条件)
## Goal(目标)
(一句话说清想要什么效果)
## Acceptance criteria(验收标准)
- (使用什么命令可以校验结果)
## Constraints(约束)
- 不泄露 secrets
- 最小化改动
## Delivery(必须交付)
1) 计划
2) 代码改动
3) 关键 diff
4) 验证通过的输出
```

## 四、角色分工

### Scout(侦察)
- 只找线索,不改代码
- 交付:文件路径清单(按重要性排序)、相似实现、约束条件、最小改动范围
- 禁止:写代码、大范围重构

### Builder(实现)
- 只做实现,最小改动
- 红线:先亮 3-7 步计划;编码后交 diff;提供测试输出;新库或超 5 文件要停;幂等
- 禁止:修改测试基线、重构无关代码

### Verifier(审查)
- 只做审查,不写实现
- 交付:风险清单、未覆盖边界、逐条 diff 结论、补救建议
- 禁止:凭感觉输出、没验证时宣称"应该没问题"

### 流转路径
```
scout.pass → builder
builder.pass → verifier
verifier.fail → builder(打回原因直接发还)
needs-human → 人工仲裁
```

## 五、排障流程卡(Bugfix Skill)

### 必须交付
1. 最小复现(可执行命令/步骤)
2. 根因定位(引用代码行号 + 日志证据)
3. 修复方案(解释为什么这样改)
4. 修复 diff
5. 验证输出(测试日志)
6. 回归测试(写不了说明技术原因)

### 卡住了怎么办
- 没法复现:列"需要补充的信息清单"
- 偶现失败:逐一排查(依赖版本/时区/网络/并发/随机种子)
- 连续 2 次失败:停手汇报

## 六、安全底线

- **最小权限**:只给特定目录访问,只开必要端口
- **写入限制**:必须写文件时,限制在 src 或 tmp 下
- **可追溯**:能看到调了什么、改了哪几行
- **重试克制**:别让重试变成死循环
- **SQL 参数化**:禁止字符串拼接 SQL
- **Secret 管理**:.env 加进 .gitignore,永不提交

## 七、流水线三关卡

1. **静态检查**:lint + 类型检查,低级错误直接拦掉
2. **业务逻辑**:跑涉及的单测/集成测试,确认功能没坏
3. **安全扫描**:硬编码密钥?高危依赖?发现了就打断

关卡没过就退回重来。

## 八、多 Agent 并发保护

- 同一时刻只允许一个 Builder 有写权限,其他只读
- 写入前必须同步最新上游状态
- 文件锁:flock 加锁
- Git 保护:每次写入前 git pull --rebase,有冲突就中断

## 九、长任务治理(ULW 循环推进)

```
每轮只盯一个改动点,动手前先说清楚验证标准
每轮结束必须交证据:改动概要 + diff + 验证日志
硬性上限:最多试 N 轮或花 T 分钟,超了停下汇报
碰红线立刻停手汇报
交接时精简上下文:3 句话压缩,不搬大段日志
```

### 大仓库三轮法
1. 第一轮:只侦察,摸清模块和接口,不写代码
2. 第二轮:做最小化修复
3. 第三轮:跑全量验证,抓漏掉的边界

## 十、评测与可观测

### 回归测试集建议
- 3 个有代表性的 Bugfix
- 3 次 Feature 增量
- 2 次重构
- 2 次补核心链路测试

### Tracing(日志规范)
每条日志必须带 `trace_id`:
```json
{"trace_id":"bugfix-123","ts":"2026-07-15T10:00:00Z","tool":"bash","input_summary":"pytest","output_summary":"passed","duration_ms":1234,"error":null}
```

### 4 个关键指标
1. 各类任务的通过率
2. 单次 Bug 修复的端到端耗时
3. 每个环节的 Token 消耗
4. 失败时卡在哪一环

## 十一、上下文文件体系

### AGENTS.md(核心)
- 项目怎么跑
- 什么能改/不能改
- 怎么验收

### .context/(跨设备同步)
- SUMMARY.md:项目概述与工作流
- task_progress.md:开发进展日志
- sync_meta.json:同步元数据
- .gitattributes:合并策略(last-push wins)

### skills/(流程卡)
- 独立文件存放,AGENTS.md 中引用
- SKILL.md 越短越好,详细内容放 references/

## 十二、新手工作流(每次写需求带这 4 句)

1. 先解释你要怎么做
2. 再改代码
3. 给我看 diff
4. 跑测试/构建

## 十三、关键心法

1. **除非在 plan mode,永远不要纠正 AI**。做错了直接 clear + git reset → 修改 prompt
2. **永远不要使用 /compact**。context 满了说明需求太大,拆成两个
3. **一个需求三轮对话还没完成,重新来**。context 用到 60% 就差不多了
4. **Agent 最关键的不是模型,而是规则 + 审核方式**
