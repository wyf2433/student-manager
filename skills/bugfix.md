# Skill: Bugfix(标准排障流程卡)

> 遇到 Bug 时,严格遵循本流程卡片。每一步都必须交付证据。

## Inputs required(初始信息)

提 Bug 时必须提供:
- 复现步骤或触发崩溃的命令
- 报错日志 / Trace
- 你怀疑的影响范围(可选)

## Outputs required(必须交付)

### 1. 最小复现
- 提取最小规模的复现逻辑
- 提供可执行的命令或步骤
- 排除无关变量

### 2. 根因定位
- 必须引用具体的代码行号(`file_path:line_number`)
- 必须引用日志证据
- 解释为什么会出错

### 3. 修复方案
- 解释为什么要这样改
- 说明是否影响其他模块
- 标注 [观察] 或 [推断]

### 4. 修复 diff
- 贴出关键 diff
- 最小化改动,不重构无关代码

### 5. 验证输出
- 执行自动化测试,贴上输出日志
- 手动验证复现步骤是否不再报错

### 6. 回归测试
- 补充回归测试代码
- 如果客观上没法写,说明技术原因

## Failure strategy(卡住了怎么办)

### 没法复现
- 列一份"需要补充的信息清单"
- 请用户提供更多上下文(环境/版本/操作序列)

### 偶现失败(Flaky test)
- 逐一排查变量:
  - 依赖版本
  - 时区
  - 网络
  - 并发
  - 随机种子

### 连续 2 次排错失败
- **停手汇报**(Stop condition)
- 不要硬撑,避免越改越乱

## 排障流程(门控机制)

```
收到Bug报告
    │
    ▼
[门1] 能复现? ──否──→ 列补充信息清单,等待
    │是
    ▼
[门2] 定位根因? ──否──→ 连续2次失败? → 停手汇报
    │是              │否→ 继续排查
    ▼
[门3] 提出修复方案
    │
    ▼
[门4] 出 diff
    │
    ▼
[门5] 跑测试 ──失败──→ 回到门3
    │通过
    ▼
[门6] 补回归测试
    │
    ▼
[完成] 交付全部证据
```

## 验收清单(强制)

```text
请你严格按照以下清单进行交付,并且每一项都必须附上证明:
1) 提取出最小规模的复现逻辑(提供可执行的命令或步骤)。
2) 确诊根因所在(必须引用具体的代码行号或日志证据)。
3) 提出修复方案,并解释你为什么要这样改。
4) 给出修复后的对比 diff。
5) 执行自动化测试并直接贴上输出日志。
6) 补充一段回归测试代码(如果客观上没法写,请写明技术原因)。
```

> 用词应为"必须交付的硬性证据",而不是"建议"或"最好"。
